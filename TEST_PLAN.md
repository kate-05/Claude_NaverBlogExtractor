# Naver Blog Crawler - 테스트 계획서

---

## 사전 준비

```bash
cd naver_blog_crawler
pip install -r requirements.txt
```

- 테스트용 네이버 블로그 2~3개 준비 (공개 블로그, 글 수가 적은 블로그 권장)
- 각 테스트 시작 전 `naver_blog_crawler.db`, `crawl_progress.json` 삭제하여 초기 상태에서 시작

---

## TC-01. 액세스 코드 검증

### TC-01-1. 유효한 코드 입력
- **조작**: 앱 실행 → 인증 다이얼로그에 `KATE2026Q1` 입력 → "확인" 클릭
- **기대**: 다이얼로그 닫힘, 메인 윈도우 표시
- **확인**: 윈도우 타이틀 "네이버 블로그 크롤러" 표시

### TC-01-2. 소문자 코드 입력
- **조작**: `kate2026q1` 입력 → "확인"
- **기대**: 대문자 변환되어 정상 인증 (다이얼로그 닫힘)

### TC-01-3. Enter 키로 인증
- **조작**: 코드 입력 후 Enter 키 누름
- **기대**: "확인" 버튼 클릭과 동일한 동작

### TC-01-4. 잘못된 코드 입력
- **조작**: `INVALID123` 입력 → "확인"
- **기대**: 빨간색 텍스트 "유효하지 않은 코드입니다" 표시, 다이얼로그 유지

### TC-01-5. 빈 코드 입력
- **조작**: 아무것도 입력하지 않고 "확인" 클릭
- **기대**: "코드를 입력해주세요" 표시

### TC-01-6. 만료된 코드 입력
- **조작**: config.py에서 임시로 만료일을 과거 날짜로 변경 → 해당 코드 입력
- **기대**: "만료된 코드입니다 (만료일: YYYY-MM-DD)" 표시
- **후처리**: config.py 원복

### TC-01-7. 다이얼로그 X 버튼 닫기
- **조작**: 코드 입력 없이 X 버튼 클릭
- **기대**: `verify_access()` 가 False 반환, 앱 종료됨 (메인 윈도우 미표시)

## TC-02. 블로그 추가

### TC-02-1. 기본 URL 형식으로 추가
- **조작**: URL 입력란에 `https://blog.naver.com/{테스트블로그ID}` 입력 → "추가" 클릭
- **기대**:
  - "추가" 버튼 일시 비활성화
  - 로그에 `[HH:MM:SS] 블로그 정보 가져오는 중: {블로그ID}` 표시
  - 블로그 목록에 새 항목 추가 (체크박스 + 블로그이름 + "대기중 (N개)" + X버튼)
  - 로그에 `블로그 추가됨: {블로그이름}` 표시
  - "추가" 버튼 다시 활성화
  - URL 입력란 비워짐

### TC-02-2. 모바일 URL 형식
- **조작**: `https://m.blog.naver.com/{블로그ID}` 입력 → "추가"
- **기대**: TC-02-1과 동일하게 정상 추가

### TC-02-3. PostList URL 형식
- **조작**: `https://blog.naver.com/PostList.naver?blogId={블로그ID}` 입력 → "추가"
- **기대**: 정상 추가

### TC-02-4. PostView URL 형식
- **조작**: `https://blog.naver.com/PostView.naver?blogId={블로그ID}&logNo=123456` 입력 → "추가"
- **기대**: 블로그 ID 추출하여 정상 추가

### TC-02-5. Enter 키로 추가
- **조작**: URL 입력 후 Enter 키 누름
- **기대**: "추가" 버튼 클릭과 동일한 동작

### TC-02-6. 잘못된 URL 입력
- **조작**: `https://www.google.com` 입력 → "추가"
- **기대**: 경고 다이얼로그 "올바른 네이버 블로그 URL을 입력해주세요." 표시

### TC-02-7. 빈 입력
- **조작**: 아무것도 입력하지 않고 "추가" 클릭
- **기대**: 아무 동작 없음 (에러 없이 무시)

### TC-02-8. 공백만 입력
- **조작**: `   ` (공백) 입력 → "추가"
- **기대**: 빈 입력과 동일하게 무시

### TC-02-9. 중복 블로그 추가
- **조작**: 이미 추가된 블로그의 URL을 다시 입력 → "추가"
- **기대**: 로그에 "블로그가 이미 존재합니다." 표시, 목록에 중복 추가 안됨

### TC-02-10. 존재하지 않는 블로그
- **조작**: `https://blog.naver.com/this_blog_does_not_exist_xyz123` 입력 → "추가"
- **기대**: 로그에 "블로그 정보를 가져올 수 없습니다" 표시 또는 ID를 이름으로 사용하여 추가

### TC-02-11. 추가 중 연속 클릭 방지
- **조작**: URL 입력 → "추가" 클릭 → 즉시 다시 "추가" 클릭
- **기대**: 첫 번째 클릭 후 버튼 비활성화 상태이므로 두 번째 클릭 무시

## TC-03. 블로그 목록 관리

### TC-03-1. 체크박스 선택
- **조작**: 블로그 항목의 체크박스 클릭
- **기대**: 체크 표시 활성화, 내부 `selected_blogs` 에 해당 ID 추가

### TC-03-2. 체크박스 해제
- **조작**: 선택된 체크박스 다시 클릭
- **기대**: 체크 해제, `selected_blogs`에서 제거

### TC-03-3. 블로그 삭제 - 확인
- **조작**: 블로그 항목의 "X" 버튼 클릭 → 확인 다이얼로그에서 "예" 선택
- **기대**:
  - 다이얼로그 "이 블로그와 모든 관련 데이터를 삭제하시겠습니까?" 표시
  - "예" 클릭 후 목록에서 해당 항목 제거
  - 로그에 "블로그가 삭제되었습니다." 표시
  - DB에서 해당 블로그 + 관련 글/댓글/공감/진행상황 모두 삭제

### TC-03-4. 블로그 삭제 - 취소
- **조작**: "X" 버튼 클릭 → "아니오" 선택
- **기대**: 목록 변경 없음, DB 변경 없음

### TC-03-5. 크롤링 중 삭제 시도
- **조작**: 크롤링 진행 중 "X" 버튼 클릭
- **기대**: 경고 다이얼로그 "크롤링 중에는 블로그를 삭제할 수 없습니다." 표시

### TC-03-6. 앱 재시작 시 목록 복원
- **조작**: 블로그 2~3개 추가 → 앱 종료 → 앱 재시작
- **기대**: 이전에 추가한 블로그가 목록에 모두 표시됨

## TC-04. 크롤링 시작

### TC-04-1. 선택한 블로그 크롤링
- **조작**: 블로그 2개 중 1개만 체크 → "시작" 클릭
- **기대**:
  - 버튼 상태 변경: 시작(비활성), 일시정지(활성), 중단(활성), 추가(비활성)
  - 선택한 블로그만 크롤링 시작
  - 해당 블로그 상태 "진행중"으로 변경
  - 로그에 단계별 메시지 표시

### TC-04-2. 전체 블로그 크롤링 (미선택)
- **조작**: 아무것도 선택하지 않고 "시작" 클릭
- **기대**: PENDING/IN_PROGRESS 상태인 모든 블로그 크롤링 시작

### TC-04-3. 크롤링할 블로그 없음
- **조작**: 목록이 비어있거나 모두 COMPLETED 상태에서 "시작" 클릭
- **기대**: 알림 다이얼로그 "크롤링할 블로그가 없습니다." 표시

### TC-04-4. 크롤링 중 "시작" 재클릭
- **조작**: 크롤링 진행 중 "시작" 버튼 클릭
- **기대**: 버튼이 비활성 상태이므로 무시됨

## TC-05. 크롤링 단계별 동작

### TC-05-1. BLOG_INFO 단계
- **조작**: 크롤링 시작 후 첫 번째 단계 관찰
- **기대**:
  - 현재 작업: "현재 작업: 블로그 정보 수집"
  - 블로그 이름/저자 이름이 DB에 업데이트됨
  - 로그에 "블로그 정보 수집 완료: {블로그이름}" 표시

### TC-05-2. POST_LIST 단계
- **조작**: BLOG_INFO 완료 후 관찰
- **기대**:
  - 현재 작업: "현재 작업: 글 목록 수집"
  - 로그에 "글 목록 가져오는 중..." → "글 목록: N개 수집됨 (페이지 X)" → "총 N개 글 목록 수집 완료"
  - 로그에 "글 N개 추가됨" 표시
  - 블로그 항목의 글 수 업데이트

### TC-05-3. POST_CONTENT 단계
- **조작**: POST_LIST 완료 후 관찰
- **기대**:
  - 현재 작업: "현재 작업: 글 내용 수집"
  - 블로그 항목 상태: "진행중 (1/N)" → "진행중 (2/N)" → ...
  - 진행률 바 점진적 증가
  - 각 글의 title, content, category, post_date가 DB에 저장됨

### TC-05-4. REACTIONS 단계
- **조작**: POST_CONTENT 완료 후 관찰
- **기대**:
  - 현재 작업: "현재 작업: 공감 수집"
  - 블로그 항목 상태: "공감 (1/N)" → "공감 (2/N)" → ...
  - 각 글의 공감 종류별 수가 reactions 테이블에 저장됨
  - sympathy_count 필드 업데이트됨

### TC-05-5. COMMENTS 단계
- **조작**: REACTIONS 완료 후 관찰
- **기대**:
  - 현재 작업: "현재 작업: 댓글 수집"
  - 블로그 항목 상태: "댓글 (1/N)" → "댓글 (2/N)" → ...
  - 댓글과 대댓글이 comments 테이블에 저장됨
  - is_reply=1인 댓글의 parent_id가 올바른 부모 댓글을 가리킴
  - like_count 필드에 좋아요 수 저장됨

### TC-05-6. 크롤링 완료
- **조작**: 모든 단계 완료 관찰
- **기대**:
  - 로그에 "블로그 크롤링 완료: {블로그이름}" 표시
  - 블로그 항목 상태: "완료 (N개)"
  - 버튼 상태 복원: 시작(활성), 일시정지(비활성), 중단(비활성), 추가(활성)
  - 현재 작업: "현재 작업: 대기 중"
  - 진행률: 100%

## TC-06. 크롤링 일시정지 / 중단

### TC-06-1. 일시정지
- **조작**: POST_CONTENT 단계 중 "일시정지" 클릭
- **기대**:
  - 로그에 "일시정지 요청됨. 현재 작업 완료 후 중단됩니다..." 표시
  - 일시정지 버튼 비활성화
  - 현재 처리 중인 글 완료 후 루프 중단
  - 'crawl_finished' 메시지로 UI 복원
  - `crawl_progress.json`에 현재 진행 상태 저장됨

### TC-06-2. 중단
- **조작**: 크롤링 중 "중단" 클릭
- **기대**:
  - 로그에 "중단 요청됨. 현재 작업 완료 후 중단됩니다..." 표시
  - 일시정지/중단 버튼 모두 비활성화
  - 현재 처리 중인 작업 완료 후 루프 중단
  - 블로그 상태 IN_PROGRESS 유지

### TC-06-3. 중단 후 재시작
- **조작**: TC-06-2 이후 "시작" 클릭
- **기대**: IN_PROGRESS 상태인 블로그에 대해 크롤링 재시작 (처음부터)

## TC-07. 이어하기 (Resume)

### TC-07-1. 앱 재시작 시 이어하기 다이얼로그
- **조작**: 크롤링 중 앱 강제 종료 → 앱 재시작
- **기대**:
  - 500ms 후 다이얼로그: "이전에 중단된 작업이 있습니다. 이어서 진행하시겠습니까?"

### TC-07-2. 이어하기 수락
- **조작**: TC-07-1 다이얼로그에서 "예" 선택
- **기대**:
  - 이미 완료된 단계(BLOG_INFO, POST_LIST 등) 건너뜀
  - POST_CONTENT 단계의 경우 `current_post_index`부터 재개
  - 나머지 미완료 단계 순차 실행

### TC-07-3. 이어하기 거부
- **조작**: TC-07-1 다이얼로그에서 "아니오" 선택
- **기대**: 다이얼로그 닫힘, 앱 대기 상태 유지, 사용자 수동 조작 대기

### TC-07-4. 진행 파일 손상 시
- **조작**: `crawl_progress.json` 내용을 깨진 텍스트로 변경 → 앱 시작
- **기대**: JSON 파싱 실패 → 기본값 `{blogs: []}` 로드 → 이어하기 다이얼로그 미표시

### TC-07-5. 진행 파일 삭제 시
- **조작**: `crawl_progress.json` 삭제 → 앱 시작
- **기대**: 파일 없음 → 기본값 로드 → 이어하기 다이얼로그 미표시

### TC-07-6. 복수 블로그 이어하기
- **조작**: 블로그 2개를 동시에 크롤링하다 중단 → 재시작 → "예"
- **기대**: IN_PROGRESS 상태인 블로그 모두에 대해 이어하기 수행

## TC-08. 내보내기

### TC-08-1. JSON 내보내기
- **조작**: 크롤링 완료된 블로그 존재 → "내보내기" → "예" (JSON)
- **기대**:
  - `exports/` 폴더에 `{블로그이름}_{타임스탬프}.json` 파일 생성
  - JSON 구조 확인:
    ```json
    {
      "blog": {"id": "...", "blog_name": "...", "author_name": "...", "url": "..."},
      "posts": [
        {
          "title": "...", "content": "...", "category": "...",
          "reactions": [{"reaction_type": "...", "count": N}],
          "comments": [
            {
              "author": "...", "content": "...", "like_count": N,
              "replies": [{"author": "...", "content": "...", "like_count": N}]
            }
          ]
        }
      ]
    }
    ```
  - 댓글과 대댓글이 계층적으로 중첩됨

### TC-08-2. CSV 내보내기
- **조작**: "내보내기" → "아니오" (CSV)
- **기대**:
  - `exports/{블로그이름}_posts_{타임스탬프}.csv` 생성
    - 헤더: id, blog_id, title, content, category, post_url, post_date, comment_count, sympathy_count, ...
    - blog_name, author_name 컬럼 포함
  - `exports/{블로그이름}_comments_{타임스탬프}.csv` 생성 (댓글 있는 경우)
    - 헤더: id, post_id, parent_id, author, content, like_count, written_at, is_reply, post_title
  - CSV 파일을 Excel에서 열어 한글 깨짐 없음 확인 (UTF-8 BOM)

### TC-08-3. 완료된 블로그 없음
- **조작**: PENDING/IN_PROGRESS 블로그만 있는 상태에서 "내보내기" 클릭
- **기대**: 알림 "내보낼 수 있는 완료된 블로그가 없습니다."

### TC-08-4. 댓글 없는 블로그 CSV 내보내기
- **조작**: 댓글이 하나도 없는 블로그를 CSV 내보내기
- **기대**: posts CSV만 생성, comments CSV는 미생성

### TC-08-5. 복수 블로그 내보내기
- **조작**: COMPLETED 블로그 2개 이상 존재 → 내보내기
- **기대**: 각 블로그별로 별도의 파일 생성

### TC-08-6. 특수 문자가 있는 블로그 이름
- **조작**: 블로그 이름에 `/`, `\` 등 포함된 경우
- **기대**: `_`로 치환되어 유효한 파일명으로 저장

## TC-11. 데이터 무결성

### TC-11-1. 댓글-대댓글 관계
- **조작**: 크롤링 완료 후 DB 직접 조회
- **검증**:
  ```sql
  -- 대댓글의 parent_id가 실제 존재하는 댓글 ID인지 확인
  SELECT c1.id, c1.parent_id
  FROM comments c1
  WHERE c1.is_reply = 1
    AND c1.parent_id NOT IN (SELECT id FROM comments);
  ```
- **기대**: 결과 0건 (모든 parent_id가 유효)

### TC-11-2. 공감 데이터 정합성
- **조작**: DB 조회로 reactions 테이블 확인
- **검증**:
  ```sql
  -- 공감 종류별 합계와 posts.sympathy_count 비교
  SELECT p.id, p.sympathy_count, COALESCE(SUM(r.count), 0) as reaction_sum
  FROM posts p
  LEFT JOIN reactions r ON p.id = r.post_id
  GROUP BY p.id
  HAVING p.sympathy_count != reaction_sum;
  ```
- **기대**: sympathy_count가 reactions 합계와 일치하거나, 공감 API 특성상 차이 허용

### TC-11-3. 글-블로그 관계
- **조작**: DB 조회
- **검증**:
  ```sql
  -- 모든 posts가 유효한 blog_id를 가지는지
  SELECT p.id FROM posts p
  WHERE p.blog_id NOT IN (SELECT id FROM blogs);
  ```
- **기대**: 결과 0건

### TC-11-4. 블로그 삭제 후 연관 데이터
- **조작**: 블로그 삭제 후 DB 조회
- **검증**:
  ```sql
  -- 삭제된 blog_id를 참조하는 데이터 없는지
  SELECT COUNT(*) FROM posts WHERE blog_id = '{삭제된ID}';
  SELECT COUNT(*) FROM progress WHERE blog_id = '{삭제된ID}';
  ```
- **기대**: 모든 쿼리 결과 0건

## TC-12. 에러 처리 및 엣지 케이스

### TC-12-1. 네트워크 끊김

- **조작**: 크롤링 후 comments 테이블의 content 확인
- **기대**: `<div>`, `<span>` 등 HTML 태그가 제거된 순수 텍스트
